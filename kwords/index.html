traditional VCS versus DVCS

A traditional version control system is centralised, and looks like this:

The "VCS metadata" is whatever the VCS needs to keep track of what you checked out last etc., and is usually part of the working directory.

The working directory of course is where you edit your programs and compile, test, etc., before committing your changes.

In git you have a copy of the repo on your local workstation. (In git this is inside the ".git" directory at the root of the working directory.)

Notice the differences:

  • You have a full-blown repo in between, so your working directory now checks out from and commits to this one.

  • Your local repo then talks to the remote repo, and it uses some new commands to do so.

what's a bare repo

A "bare" repo has no working tree, like the one on the right side. This is the only kind of repo you should push to.

When you push, git can only update the blue part. If you somehow manage to push to a non-bare repo, the pink part may go out of sync with the blue.

The correct way to update the working directory is for a developer to actively pull from the server repo. Since they're "local", they can then resolve conflicts, if any.

Side note: by convention, bare repos have directory names ending in .git. So your server side bare repo could be in $HOME/repositories/my-proj.git.
